# 编码助手配置

## 项目定位

这个目录用于创建和修改「赢单」应用的 UI 原型界面。

「赢单」是一个面向外贸业务场景的成交顾问应用，核心体验是：用户选择一个外贸业务场景，粘贴客户信息、询盘、谈判背景或业务目标，然后由 AI 生成调研、话术、邮件、会议、谈判和成交建议。

本目录不是正式生产代码仓库，也不是产品帮助中心、功能说明文档或教学页面。它完全用于创建可以直接打开、演示和讨论的 HTML UI 原型。后续新增功能时，应先在这里做出初版 UI 图和用户流程，用来判断：

- 需要哪些功能模块。
- 用户从哪个入口进入任务。
- 每个界面需要收集哪些信息。
- 不同界面之间如何跳转。
- 哪些功能适合合并，哪些功能需要拆成独立页面。

重要强调：

- 原型界面必须像真实产品一样可操作，不要在页面里写“这个功能用于……”“使用方法是……”“后续正式产品……”这类功能讲解内容。
- 需要说明设计意图、技术选择或后续开发注意事项时，写在 `CONTEXT.md`、代码注释或聊天回复里，不要写进用户会看到的原型界面。
- 页面文案只保留真实用户在产品里会看到并使用的内容，例如标题、字段标签、按钮、占位提示、筛选项、结果示例和业务数据。

## 技术栈约定

- 默认以前端静态原型为主：`HTML` + `CSS` + 原生 `JavaScript`。
- 单页原型优先放在一个 `.html` 文件里，方便浏览器直接打开和演示。
- 只有当交互明显复杂、状态很多、需要组件复用时，才考虑引入框架。
- 创建正式前后端项目或复杂功能前，必须先询问用户：
  - 前端使用什么语言/框架？
  - 后端使用什么语言/框架？
  - 同时用简洁中文说明不同技术选择的利弊。

### 技术选择利弊

静态 HTML 原型：

- 优点：启动快、文件少、容易改、可以直接打开给人看。
- 缺点：不适合复杂状态管理，不适合长期维护大型应用。

原生 JavaScript：

- 优点：不用构建工具，适合快速验证交互。
- 缺点：功能变多后容易散，需要更认真地组织代码和注释。

React/Vue 等框架：

- 优点：适合复杂页面、复用组件、多状态交互。
- 缺点：需要构建环境，原型交付成本更高。

Python 后端：

- 优点：适合快速接 API、处理文件、生成数据和调试 AI 工作流。
- 缺点：如果只是做 UI 原型，会显得过重。

## 功能区域命名约定

后续讨论和修改需求时，统一使用下面这套命名，避免“页面、模块、区域”混用导致定位不清。

### 一级功能区

一级功能区是左侧主导航里的大入口，代表用户来这里要做哪类事。

- `问一下`：通用外贸问答。
- `销售准备`：成交前资料准备、公司/产品/资产/SOP。
- `成交顾问`：围绕具体客户和成交动作的顾问。
- `技能Skill`：更细的外贸工作流工具。
- `客户Kass`：客户档案/客户上下文。
- `历史记录`：历史会话和历史任务。

### 二级模块

二级模块是一级功能区下面的具体模块。

`销售准备` 下的二级模块：

- `了解公司`
- `了解产品和市场`
- `公司资产`
- `外贸流程`

`成交顾问` 下的二级模块：

- `客户背调顾问`
- `场景谈判顾问`
- `询盘分析回复`

`技能Skill` 下的二级模块：

- `市场调研`
- `新客开发信`
- `客诉处理`
- `客户激活`
- `关系维护`
- `海外电销`
- `视频会议`
- `地推陌拜`
- `来访接待`
- `展会成交`
- `谈判`
- `组合标题`

### 子流程 / 子页面

子流程或子页面是某个二级模块里继续展开的具体分支。

示例：

- `技能Skill > 展会成交 > 展前筹备顾问`
- `技能Skill > 展会成交 > 展中客户接待`
- `技能Skill > 展会成交 > 展后跟进拜访`
- `技能Skill > 组合标题 > 标题组合`
- `技能Skill > 组合标题 > 标题+卖点+FAQ`
- `销售准备 > 了解产品和市场 > 产品目录表`
- `销售准备 > 了解产品和市场 > 产品营销海报库`
- `销售准备 > 外贸流程 > 外贸全流程 SOP 导图`

### 页面区域

右侧主内容区域统一叫 `工作区`。用户看内容、填信息、点按钮、看结果的地方都属于工作区。

工作区内的固定区域命名：

- `输入区`：大文本框、附件、模型选择、语音、发送按钮。
- `筛选条件区`：国家、客户类型、谈判场景、开发目的等 chip 或下拉项。
- `推荐动作区`：输入区下方的快捷任务卡片，例如客户画像、报价邮件、现场话术。
- `流程说明区`：用 1、2、3 卡片说明这个功能怎么走的区域。
- `结果预览区`：点击发送或 AI 生成后出现的结果内容。
- `抽屉`：右侧滑出的面板，例如历史、教学视频。
- `弹窗`：居中弹出的面板，例如附件、账号设置、客户编辑。

### 需求描述格式

后续需求优先按这个格式描述：

`一级功能区 > 二级模块 > 子流程/子页面 > 具体区域`

示例：

- `销售准备 > 了解公司 > 输入区`
- `销售准备 > 了解产品和市场 > 产品目录表`
- `技能Skill > 展会成交 > 展中客户接待 > 结果预览区`
- `成交顾问 > 询盘分析回复 > 筛选条件区`

实现或修改时，先按这套路径定位，再动对应代码和界面。

## 扣子工作流资料库维护规则

项目根目录的 `coze-workflows/` 用于统一记录赢单功能对应的扣子工作流资料。

当用户提供扣子工作流的调用链接、调用函数、input schema 或 output schema 时，优先写入 `coze-workflows/`，不要写进 `index.html` 原型界面。

当用户提供扣子节点画布、节点样例或要求“下一次直接按这些节点做工作流”时，优先写入 `coze-workflows/_node-reference/`，并维护节点类型索引。后续搭建具体业务工作流前，先读取该目录，再根据用户需求选择最短节点链路。

每个工作流必须单独建目录，建议格式：

```text
coze-workflows/<一级功能区>-<二级模块>-<工作流名>/
```

每个工作流目录应包含：

- `workflow.md`：记录业务用途、赢单功能路径、调用链接、调用函数摘要、schema 摘要、字段映射、示例、异常和版本记录。
- `input.schema.json`：保留扣子提供的原始 input schema。
- `output.schema.json`：保留扣子提供的原始 output schema。
- `call-function.md`：记录调用函数原文、语言、依赖、鉴权占位和调用注意事项。
- `imagegen-brief.md`：记录后续 ImageGen 图片的图示目标、字段来源、提示词和生成记录。

维护要求：

- schema 原文必须完整保留，不要随意改字段名、字段类型和必填规则。
- output schema 未提供时，只能明确写 `待提供`，不能编造返回字段。
- 调用函数中的真实 Token、Cookie、密码、账号密钥必须替换为占位符。
- 不要写入真实客户资料、真实聊天记录、手机号、邮箱或其他隐私。
- 用户明确要求生成图时，直接使用 ImageGen tool，并把图片需求、提示词和对应调用函数关系记录到 `imagegen-brief.md`。
- 如果用户只提供部分资料，就先记录已提供内容，缺失项标记为 `待提供`。

## 核心页面与业务含义

### 问一下

通用外贸问答入口。用户可以输入外贸相关问题，例如新手外贸准备、运营动作、业务策略等。

原型重点：

- 输入框要清楚表达「可以问任何外贸相关问题」。
- 不需要太多前置表单，保持轻量。
- 适合作为默认首页或兜底入口。

### 客户背调顾问

用于根据客户所在国家/地区、行业、标签、公司名称、官网链接等信息做客户背景调研。

原型重点：

- 输入信息应围绕客户身份和公司背景。
- 可预留输出区：客户画像、采购可能性、风险点、沟通建议、下一步动作。

### 场景谈判顾问

用于选择常见谈判场景，并结合用户补充的问题生成谈判策略。

原型重点：

- 需要有「谈判场景」选择控件。
- 页面应突出场景选择 + 问题详情两个输入来源。
- 输出可围绕报价、让步、催回复、客户异议、付款条件等展开。

### 询盘分析回复

用于粘贴客户询盘或聊天记录，分析询盘质量并生成回复建议。

原型重点：

- 文本框应支持长文本。
- 可以提示用户补充产品信息、价格区间、底线要求。
- 输出可包含客户诚意判断、关键信息缺口、回复策略、英文邮件草稿。

### 市场调研

用于根据核心产品、目标国家/地区、目标客户类型做市场调研和选品建议。

原型重点：

- 需要国家/地区和客户类型筛选。
- 输出可包含市场机会、客户类型、渠道建议、竞品风险、切入话术。

### 新客开发信

用于生成首次接触、报价、催回复等外贸开发信。

原型重点：

- 需要国家、客户类型、开发目的等选项。
- 要支持语言选择，例如英文、西班牙语等。
- 输出要像真实邮件，不要只给泛泛建议。

### 客诉处理

用于分析客户投诉、订单问题、责任边界和赔付方案。

原型重点：

- 需要收集客户信息、订单信息、问题证据、客户诉求、我方底线。
- 可以有客诉内容、客户类型、客户关系等下拉项。
- 输出应包含责任判断、沟通策略、英文回复、赔付建议。

### 客户激活

用于唤醒沉默客户或老客户。

原型重点：

- 需要沉默时长、历史合作情况、新卖点或优惠信息。
- 输出应包含唤醒邮件、后续跟进节奏和不同反应下的应对。

### 关系维护

用于节日、生日、项目节点等轻商务关系维护。

原型重点：

- 需要客户文化背景、关系程度、时间节点和沟通目的。
- 输出应短、自然、不过度销售。

### 海外电销

用于生成外贸电话沟通脚本。

原型重点：

- 需要国家/时区、客户关系、沟通目的和常见异议。
- 输出应包含开场白、关键问题、异议处理、结束语和跟进动作。

### 视频会议

用于整理视频会议发言、议程和纪要。

原型重点：

- 需要参会方、会议主题、沟通目标、议程或资料链接。
- 输出可包含会议前准备、发言稿、会议纪要和会后跟进邮件。

### 地推陌拜

用于线下拜访前做客户摸底和话术准备。

原型重点：

- 需要客户国家/地区、行业、公司名、官网或地图链接。
- 输出可包含拜访目标、开场话术、观察清单和现场转化动作。

### 来访接待

用于客户来厂、验厂、考察等接待方案。

原型重点：

- 需要客户国家、行业、公司、来访目的。
- 输出可包含接待流程、现场介绍话术、风险点和会后跟进建议。

### 展会成交

包含展前、展中、展后相关子场景。

已观察到的子入口：

- `展前筹备顾问`
- `展中客户接待`
- `展后跟进拜访`

原型重点：

- 展会类页面适合用顶部子标签切换。
- 展前关注准备清单、邀约话术和分工。
- 展中关注客户接待、名片收集、意向判断和现场推进。
- 展后关注分层跟进、拜访安排、邮件节奏和成交推进。

### 谈判

用于更通用的谈判输入和策略生成。

原型重点：

- 当前页面输入提示较少，后续原型应补充谈判背景、客户异议、我方目标、底线和可让步条件。

### 组合标题

包含标题生成和标题扩展类子场景。

已观察到的子入口：

- `标题组合`
- `标题+卖点+FAQ`

原型重点：

- 支持批量粘贴关键词。
- 输出应适合电商或外贸商品内容，包括标题、卖点、FAQ。

## 原型设计规则

- 第一屏直接展示可操作界面，不做营销式落地页。
- 优先复刻赢单的真实信息架构：左侧导航 + 右侧任务工作区。
- 原型要能表达用户流程，不只是静态好看的图。
- 原型不是功能讲解稿，界面里不要出现解释产品能力、说明设计目的、介绍开发计划或指导用户如何理解原型的文字。
- 如果某段文案不能直接帮助用户完成当前业务动作，就不要放进界面；尤其避免大段说明卡片、流程教学卡片、功能介绍卡片和“这里用于……”式占位文字。
- 每个功能最好都有对应的动画效果，用动画表达真实操作反馈、状态变化、生成过程、数据流转或页面切换，让原型更像可用产品。
- 动画必须服务功能理解和操作手感，不要做无意义装饰；优先使用轻量的 hover、active、loading、展开收起、结果生成、卡片进入和状态切换动画。
- 新增交互或动画时，只实现用户明确要求的对应流程和反馈，不要擅自增加保存、发送、生成、同步、待办等额外功能入口，避免原型交付给开发时产生多余需求坑。
- 每个页面必须能看出：
  - 当前是什么业务场景。
  - 用户要输入什么。
  - 有哪些可选条件。
  - 点击后预期会生成什么。
  - 下一步可以去哪里。
- 新增功能时，优先补齐用户流程，再考虑视觉细节。
- 不要为了炫技引入复杂动画、复杂框架或过多视觉装饰。
- 操作型界面要安静、清晰、适合反复使用。
- 卡片、按钮、输入框要有稳定尺寸，避免文字挤压和布局跳动。
- 移动端和桌面端都要检查文字不重叠、不溢出。

## 文件与目录规则

- 优先编辑现有文件。
- 如果是单个原型，优先创建一个语义清楚的 `.html` 文件。
- 如果原型包含多条用户流程，可以按功能创建多个 `.html` 文件，但不要过度拆分。
- 不主动创建新的文档文件，除非用户明确要求，或项目规则要求维护 `CONTEXT.md`。
- 本项目必须保留项目级 `CONTEXT.md`，用于说明项目目标、入口、模块、验证方式和后续改动位置。
- 创建或修改 skill 时，不需要创建、补充或更新 `CONTEXT.md`。

## 代码风格

- 代码要简洁、优雅、容易读。
- 用户不是专业开发者，代码注释必须详细清晰。
- 每个 JavaScript 函数都要写注释，说明：
  - 函数的作用。
  - 参数的含义和类型。
  - 返回值的含义。
  - 可能抛出的异常。
- 复杂逻辑必须逐步注释，解释为什么这么做。
- 关键代码行要解释作用和原因。
- 推荐使用类型提示风格的 JSDoc，让编辑器更容易理解参数和返回值。
- 不要硬编码密码、Token、Cookie、手机号、邮箱等敏感信息。

## 日志与调试

- 如果原型只有静态布局，可以不强行做复杂日志系统。
- 如果加入 JavaScript 交互，应在关键操作处使用清晰的 `console.log`、`console.warn` 或 `console.error`，方便调试。
- 如果后续增加 Python 脚本或后端服务，必须使用 Python 标准库 `logging` 模块记录关键操作、状态变化和异常信息。

## 浏览器验证

修改 HTML/CSS/JS 后，应使用浏览器打开或刷新页面验证。

重点检查：

- 页面是否能正常打开。
- 左侧导航和主工作区是否清楚。
- 按钮、输入框、下拉项是否可见。
- 主要用户流程是否能走通。
- 桌面和窄屏下文字是否重叠或溢出。
- 不要点击发送、删除、保存账号设置等可能产生真实副作用的动作，除非用户明确确认。

## 安全边界

- 不要提交真实客户资料、聊天记录、手机号、邮箱、账号信息。
- 不要复述或扩散浏览器中看到的用户历史内容。
- 不要点击删除历史、删除客户、修改账号设置等高风险操作。
- 不要在原型中写入真实 API Key 或真实登录态。
- 表单提交、发送消息、上传文件、删除数据、修改权限等动作，在真实网页里执行前必须再次确认。

## Excel 交付规则

如果任务涉及 `.xlsx` 交付，交付前必须执行固定流程：

1. 先用 LibreOffice headless 重存。
2. 清理 `xl/tables/`。
3. 清理空 `xl/drawings/`。
4. 清理 `tableParts`。
5. 清理 drawing/table relationships。
6. 通过 `unzip -t` 验证。
7. 通过 `openpyxl.load_workbook()` 验证。
8. 扫描包内 table/drawing 残留。

任一步失败都禁止交付。
